Following is a textual walk-through of the Balance Inquiry – getBalance operation. While Self Care is depicted as consumer of the service in the diagram above, it is applicable for all other consumer channels.

1.  On Startup, Balance Inquiry component invokes [findPlanOffering](../../catalog/Manage Product Offering/Product Offering Inquiry.md#findplanoffering) operation of [Product Offering Inquiry](../../catalog/Manage Product Offering/Product Offering Inquiry.md) service to fetch the Plan Offerings.  
    **Note:** specify /filterCriterias[name = 'PLANOFFER']/value = 'PLANOFFER', /filterCriterias[name = 'COMPOSITE PRICE']/value = 'COMPOSITE PRICE', /paging/pageSize = '1000', /paging/offset = '1'  
    **Note:** If one Composite Price exists for Plan Offering (planOffering/compositePrice), use as-is; If more than one Composite Price exists for Plan Offering, use the Composite Price with priceType = 2: ONETIME and priceSubType = 10: PAYABLE

2.  Balance Inquiry component caches the data elements of the Plan Offering required for enrichment.  
    **Note:** Following data elements are cached:
    -  planID
    -  planName
    -  planDesc
    -  planType
    -  planSubType
    -  planSubSubType
    -  planBillingType
    -  planBundled
    -  planStackedQueued
    -  planCategory
    -  planSubCategory
    -  planEocnApplicability
    -  businessInteractionLevel
    -  compositePrice[]
        -  priceID
        -  price
        -  priceType
        -  priceSubType

3.  Service Consumer (e.g. Self Care) invokes getBalance operation of Balance Inquiry service to determine the pre-paid balances for an account.

4.  If Account ID is not available,
    1.  Balance Inquiry component invokes [retrieveCustomerServiceConfiguration](../../inventory/Manage Customer Information/Customer Configuration Inquiry.md#retrievecustomerserviceconfiguration) operation of [Customer Configuration Inquiry](../../inventory/Manage Customer Information/Customer Configuration Inquiry.md) service to retrieve the Account Id.

5.  Balance Inquiry component retrieves the transaction routing details (Endpoint URL) for the destination instance of OCS. Refer [Routing – Outbound – Service Component to route](../../appendices/Appendix G.md#routing-outbound-service-component-to-route) for details.  
    **Note:** The Routing Zone received either in the input or retrieved via. [retrieveCustomerServiceConfiguration](../../inventory/Manage Customer Information/Customer Configuration Inquiry.md#retrievecustomerserviceconfiguration) operation of [Customer Configuration Inquiry](../../inventory/Manage Customer Information/Customer Configuration Inquiry.md) service is used, if available.

6.  Balance Inquiry component translates the request from CMM (Reliance SID) to the proprietary format of OCS and invokes chargeableItemCheckCharge operation of ChargeableItemCharging service.  
    **Note:** For any Plan Offering where data balance bucket exists with source "Transfer" (identified by /planOfferings/commonDataList/nonMonetaryBuckets/commonDataList/source = 'Transfer'), following change are made for the Plan Offering which are enabled / disabled by a configurable parameter overrideTransferPlanOffering = Y or N  
    Transfer balance bucket of non-monetary data transferred from Donor to Recipient in Family Plan Offering – Shared Plan is returned as Donor's Plan Offering applied to Recipient. This results in Recipient having multiple post-paid plan offerings returned till the transferred balance bucket is active which causes multiple issues. Such Plan Offering is overridden with Plan Offering type = 44 ensuring multiple Postpaid Plan Offerings are NOT returned for the Recipient.
    -  Replace Plan Offering Type with 44: Shared Plan; specify /planOfferings/commonDataList/type = '44'
    -  Remove Plan Offering businessInteraction Level; identified by /planOfferings/commonDataList/businessInteractionLevel
    -  Remove Plan Offering Composite Price; identified by /planOfferings/commonDataList/compositePrice

7.  The response received from OCS is translated back to the CMM (Reliance SID).

8.  If the Balance Type is not Monetary,
    1.  Balance Inquiry component retrieves the Plan Offering details from the cache for the Plan Offering(s) and enriches in the response.
    2.  If Plan Offering(s) are not retrieved from Cache,
        1.  Balance Inquiry component invokes [queryPlanOffering](../../catalog/Manage Product Offering/Product Offering Inquiry.md#queryplanoffering) operation of [Product Offering Inquiry](../../catalog/Manage Product Offering/Product Offering Inquiry.md) service to retrieve Plan Offering details and enrich the response.  
            **Note:** specify /planOffering/planID = <List of PlanOffering Id\>, /filterCriterias[name = 'RANGE_NAME'] = 'PLANOFFER', /filterCriterias[name = 'RANGE_NAME'] = 'COMPOSITE PRICE'  
            **Note:** If one Composite Price exists for Plan Offering (planOffering/compositePrice), use as-is; If more than one Composite Price exists for Plan Offering, use the Composite Price with priceType = 2: ONETIME and priceSubType = 10: PAYABLE  
            **Note:** Retrieves details of dependent plan offering as well (identified by /planOfferings/commonDataList/planOfferingDependency/id)
        2.  Balance Inquiry component also updates the cache with the Plan Offering details.

9.  The response is then returned back to the invoking consumer.
