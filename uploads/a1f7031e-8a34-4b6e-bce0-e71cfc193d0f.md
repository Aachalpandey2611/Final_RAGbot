Following is a textual walk-through of the approach to implement Refill Preference Publication:notifyRefillPreference.

1.  On Startup,
    1.  Refill Preference Publication component invokes the lookupValue operation of Reference Data Inquiry Service to retrieve the Payment Mode.  
        **Note:** specify lovType = 'PAYMENTMODE'
    2.  Refill Preference Publication component caches the lookup values required for enrichment.

2.  SAP CRM publishes an IDOC (ZCRMXIF_REFILL_PREF_IDOC) which in turn invokes notifyRefillPreference operation of Refill Preference Publication service upon registration, modification or deregistration of Refill Preference.

3.  Refill Preference Publication component translates the message from proprietary message model of SAP to CMM (Reliance SID).

4.  Refill Preference Publication component invokes findRefillPreference operation of Refill Preference Inquiry to retrieve the existing Refill Preference available for the Customer.  
    **Note:** specify /refillReference/customer/partyId = <Customer ID received in the message\>  
    **Note:** The count of the retrieved Refill Preference is used to determine the Notification template for sending notifications.

5.  If Refill Preference is for FTTx Product and Refill Preference is NOT for billing location or installation location and Refill Preference is NOT for Plan Update on Expiry (identified by /refillReference/product/identifier[name = 'R4GID'] exists and /refillReference/individual/partyRole[partyRoleId = 'CRM004' or partyRoleId = 'CRM002'] doesn't exists and /refillReference/reasonCode != 'PLAN_UPDATED_ON_EXPIRY')
    1.  Refill Preference Publication component invokes [queryCustomer](../../inventory/Manage Customer Information/Customer Inquiry.md#querycustomer) operation of [Customer Inquiry](../../inventory/Manage Customer Information/Customer Inquiry.md) service to retrieve FLN of JHV Product for sending in the Customer Notification (notification variable <<FIXED_LINE_NUMBER\>\>).  
        **Note:** specify /customerId = <from received message; /refillReference/individual/partyId\>, /filterKeyList/filterKey = <PRODUCT\>  
        **Note:** FLN is identified by /customers/products[productCode = 'P10024']/identifier[category = '4' and subCategory = '2']/value.  
        **Note:** If FLN is not retrieved, use the FTTx Product Service Id (identified by /refillReference/product/Identifier/value).

6.  Refill Preference Publication component routes the message by persisting a copy to the messaging queue for each consumers:
    1.  **EDIF:** For EDIF sync  
        **Note:** EDIF maintains the Refill Preference
        -  Delete: Action = D: Deregister or Status = R: Rejected
        -  Create: Action = R: Register
        -  Update: Action = U: Update

        **Note:** If Approved Refill Preference (identified by /refillReference/action = R: Registered and /refillReference/status = A: Approved) is received and an Approved Refill Preference exists in EDIF for the Customer,
		
        -  If the IDOC generation date time of received Refill Preference (identified by /refillReference/characteristics[attributeName='IDOC_CREATE_DATETIME']/attributeValue) is greater than existing Refill Preference,
            -  Remove the existing Refill Preference
            -  Store the received Refill Preference
        -  Otherwise
            -  Ignore the received Refill Preference

        **Note:** If Refill Preference is NOT ignored and Refill Preference is NOT for a billing location or installation location (identified by /refillReference/individual/partyRole[partyRoleId = 'CRM004' or partyRoleId = 'CRM002'] doesn't exists), EDIF Refill Preference Publication Component also maintains the status of latest Refill Preference in the Customer Space as to include it in the response of Customer Configuration Inquiry::retrieveCustomerServiceConfiguration as a Characteristic AUTOPAY_STATUS @ Customer.
		
        -  If the Refill Preference is Registered and Status is Approved (identified by /refillReference/action = R and /refillReference/status = A)

            -  Maintain the value of Refill Preference Status as A
			
        -  If the Refill Preference is Registered and Status is Pending (identified by /refillReference/action = R and /refillReference/status = P)
            -  If another Refill Preference exists with Status = A
                -  Maintain the value of Refill Preference Status as A
            -  Otherwise
                -  Maintain the value of Refill Preference Status as P
        
		-  If the Refill Preference is Registered and Status is Rejected (identified by /refillReference/action = R and /refillReference/status = R)
            -  If another Refill Preference exists with Status = A
                -  Maintain the value of Refill Preference Status as A
            -  Otherwise
                -  Remove the value of Refill Preference Status
        
		-  If the Refill Preference is Deregistered (identified by /refillReference/action = D)
            -  If the Refill Preference is removed (identified by /refillReference/characteristics[attributeName = 'SWAP_REFILL_PREF'] does not exists)
                -  If another Refill Preference exists
                    -  Maintain the value of Refill Preference Status from value of another Refill Preference
                -  Otherwise
                    -  Remove the value of Refill Preference Status

    2.  **DigitalApi Streamer:** For DigitalApi Streamer sync

    3.  **Campaign Fulfillment Platform:** For Campaign Fulfillment Platform sync  
        **Note:** Events enable providing Cashback Vouchers upon successful registration. Following is the list of scenarios which are used for actions on Cashback Vouchers:

        | Scenario | Action | Status | Characteristic: SWAP_REFILL_PREF |
        | :--- | :--- | :--- | :--- |
        | Successful Registration for AutoPay | R: Register | A: Approve | SWAP_REFILL_PREF doesn't exists |
        | Modify AutoPay amount [Modify Plan Id / Value] | U: Update | A: Approve | |
        | Modify Mandate and AutoPay amount | R: Register | A: Approve | SWAP_REFILL_PREF = Y |
        | De-Registration [Customer initiated] | D: Deregister | | SWAP_REFILL_PREF doesn't exists |

        **Note:** Campaign Fulfillment Platform maintains the Refill Preference/Plan Offering Id, Price for each Service ID and uses it to compare against the updated values provided to make necessary invocations.  
        **Note:** Cashback Vouchers are not applicable for FTTx Plan Offering hence Cashback Vouchers are not provided for Customers registering for JioFiber Product.  
        **Note:** Campaign Fulfillment Platform uses the Refill Preference for Bill Payment (identified by /refillReference/PlanOffering/id does not exists) if the Referral Code (MSISDN) of the Referee exists.  
        **Note:** Upon Successful Registration for AutoPay, the Referral Code (MSISDN) of the Referee (identified as Characteristic[Name = "REFERRAL_CODE"]/Value), if available is provided to Campaign Fulfillment Platform which use it to provide the incentives to both Referee (the one who refers) and Referrer (the one who is referred).

    4.  **Process Mandate Cancellation:** If Action = D: Deregister Refill Preference, Refill Preference Publication component persists the message to the messaging queue. The message is then consumed by the Process Mandate Cancellation to process the mandate cancellation.

    5.  **SAP CI – RJIL:** If Refill Preference is registered for Postpaid Product for CMS_CODE IN [05: Jio Money; 16: JioOnePay Configurable List] (identified by /refillReference/action = R: Register Refill Preference and /refillReference/status = P: Pending and /planOffering/id doesn't exist and /refillReference/characteristics[name = 'CMS_CODE']/value in [05, 16; Configurable List]),  
        **Note:** CMS_CODE = 16: JioOnePay Aggregator interacts with JioMoney Payment Gateway
        1.  If Refill Preference is for billing location (identified by /refillReference/individual/partyRole[partyRoleId = 'CRM004'] exists)  
            **Note:** Enterprise Product
            -  Refill Preference Publication component invokes notifyRefillPreference operation of SAP CI Enterprise Refill Preference Publication to create Mandate Details into SAP CI by persisting the message to the messaging queue.
        2.  Else If Refill Preference is for FTTx Product (identified by /refillReference/product/identifier[name = 'R4GID'] exists)  
            **Note:** Retail JioFiber Product.
            -  Refill Preference Publication component invokes notifyRefillPreference operation of SAP CI FTTx Refill Preference Publication to create Mandate Details into SAP CI by persisting the message to the messaging queue.
        3.  Otherwise (Refill Preference is for Retail Mobility),
            -  Refill Preference Publication component invokes notifyRefillPreference operation of SAP CI Refill Preference Publication to create mandate details in SAP CI by persisting the message to the messaging queue.

    6.  **Customer Notification:** If not Plan Update on Expiry (identified by /refillReference/reasonCode != 'PLAN_UPDATED_ON_EXPIRY'), send notification to the customer for registration, modification or deregistration of Refill Preference.  
        **Note:** Plan Update on Expiry is applicable when business substitutes the AutoRefill opted Plan Offering with new Plan Offering; In this scenario the notification is emitted by Process Auto Refill directly to Customer.
        1.  Refill Preference Publication component evaluates the elements from the payload of the received message, count of the retrieved Refill Preference to determine the Notification template for sending notifications. Refer to the below table to identify the various scenarios and corresponding notification.
        2.  If Refill Preference is for a billing location or installation location (identified by /refillReference/individual/partyRole[partyRoleId = 'CRM004' or partyRoleId = 'CRM002'] exist)  
            **Note:**  
            -  If Refill Preference is for billing location (identified by /refillReference/individual/partyRole[partyRoleId = 'CRM004'] exist) 
			    *  specify /notification/characteristics[name = 'EC_BL_CUST_ID']/value = <Customer Id – RJIL; identified by /refillReference/customer/customerID\>
            -  If Refill Preference is for installation location (identified by /refillReference/individual/partyRole[partyRoleId = 'CRM002'] exists) 
			    *  specify /notification/characteristics[name = 'EC_IL_CUST_ID']/value = <Customer Id – RJIL; identified by /refillReference/customer/customerID\>
            -  Refill Preference Publication invokes handleNotification operation of Enterprise Outbound Notification service (Protocol / Transport = XML/JMS) to submit a request for sending notification to the Authorized Signatories (recipient = Customer) of the Billing Location for registration or modification or de-registration of Refill Preference, by putting a message to the messaging queue.
            -  Refill Preference Publication invokes handleNotification operation of Enterprise Outbound Notification service to submit a request for sending notification to the KAM and RM (recipient = Associate) of the Billing Location for registration or modification or de-registration of Refill Preference, by putting a message to the messaging queue.

                **Note:** specify (for both notifications)  
                /notification/id = <determined Notification Template Id\>  
                /notification/interactionDateTime = <SYSDATETIME\>  
                /notification/referenceKey = <from received message; /refillReference/refillPreferenceReferenceNumber\>  
                /notification/characteristics[name = 'BLI_NAME']/value = <from received message; /refillReference/customer/organization/name\>  
                /notification/characteristics[name = 'ACCOUNT_ID']/value = <from received message; /refillReference/account/accountId\>  
                /notification/characteristics[name = 'PAYMENT_MODE']/value = <performs a lookup for the LOV PAYMENTMODE, /refillReference/autoPayment/paymentMethod\>  
                /notification/characteristics[name = 'DUE_AMOUNT']/value = <from received message; /refillReference/planOffering/compositePrice/price\>  

        3.  Otherwise
            -  Refill Preference Publication component invokes [handleNotification](../../common/Customer Communication/Customer Notification.md#handlenotification) operation of [Customer Notification](../../common/Customer Communication/Customer Notification.md) service to submit a request for sending notification to the customer, for registration or modification or de-registration of Refill Preference, by putting a message to the messaging queue.  
			    
				**Note:** Different notification templates used for registration, modification and de-registration of Refill Preference for Mobility and JioFiber Customers (identified by /product/Identifier/name = 'MSISDN' or 'R4GID').

                **Note:** specify  
                /templateID = <Identified template Id\>  
                /payloadMessage/serviceID = <from received message; /refillReference/product/Identifier/value\>  
                /payloadMessage/customerID = <from received message; /refillReference/individual/partyId\>  
                /payloadMessage/attributes/commonDataList[name = 'SERVICE_ID']/value = <from received message; /refillReference/product/Identifier/value\>  
                /payloadMessage/attributes/commonDataList[name = 'MSISDN']/value = <from received message; /refillReference/product/Identifier/value\>  
                /payloadMessage/attributes/commonDataList[name = 'PLAN_DENOMINATION']/value = <from received message; /refillReference/PlanOffering/CompositePrice/price\>  
                /payloadMessage/attributes/commonDataList[name = 'PAYMENT_MODE']/value = <performs a lookup for the LOV PAYMENTMODE, data for which is cached on startup; /refillReference/CustomerPayment/paymentMethod\>  
                /payloadMessage/attributes/commonDataList[name = 'FIXED_LINE_NUMBER']/value = <from queryCustomer response; /customers/products[productCode = 'P10024']/identifier[category = '4' and subCategory = '2']/value\>

#### Interaction Diagram

``` mermaid
sequenceDiagram
  autonumber
  participant DAPI_RPP as DigitalApi (Refill Preference Publication)
  participant MSG_SAPCI as Messaging (SAPCI.NOTIFYREFILLPREFERENCE.IN.Queue)
  participant DAPI_CIRPP as DigitalApi (SAP CI Refill Preference Publication)
  participant SAP_CI as SAP CI (ZCI_FG_AUTOPAY)

  DAPI_RPP->>MSG_SAPCI: PUT
  DAPI_CIRPP->>MSG_SAPCI: GET
  DAPI_CIRPP->>+SAP_CI: ZCI_MANDATE_STATUS_POSTPAID
  SAP_CI-->>-DAPI_CIRPP: ZCI_MANDATE_STATUS_POSTPAID
```

Following is a textual walk-through of the approach to implement SAP CI Refill Preference Publication: notifyRefillPreference.

1.  SAP CI Refill Preference Publication component (belonging to Service Translator layer – SAP CI) consumes the message from the messaging queue.

2.  SAP CI Refill Preference Publication component translates the message from CMM (Reliance SID) to the proprietary message model of SAP and invokes the SAP CI RFC ZCI_MANDATE_STATUS_POSTPAID to create mandate details in SAP CI – RJIL.  
    **Note:** The request is submitted to master instance of SAP CI – RJIL.  
    **Note:** In an event of system exceptions, DigitalApi Platform retries the message processing as per the configured policies.
