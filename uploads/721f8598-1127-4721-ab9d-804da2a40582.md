Following is a textual walk-through of the Balance Replenishment – replenishBalance service.  
**Note:** Only one Replenishment Order Line Item is supported.

1.  On Startup,

    1.  Balance Replenishment component invokes the [findPlanOffering](../../catalog/Manage Product Offering/Product Offering Inquiry.md#findplanoffering) operation of [Product Offering Inquiry](../../catalog/Manage Product Offering/Product Offering Inquiry.md) service to fetch the Plan Offerings.

    2.  Balance Replenishment component caches the data elements of the Plan Offering required for invoking recharge account and sending notifications.  
        **Note:** Plan Offering: Type, Subtype, isBundled, Description; Plan Specification: Subtype (10: Topup) indicates if monetary entitlements are available in Plan Offering.

    3.  Balance Replenishment component retrieves the LOV for MERCHANT_DESTINATION and caches the data for Merchant Code and corresponding Messaging Queue of notifyBalanceReplenishmentStatus.

2.  The consumer application invokes replenishBalance operation of Balance Replenishment service for replenishment (top-up, recharge) of pre-paid balances for an account.

3.  Balance Replenishment Proxy component performs the basic semantic validations such as compliance of Balance Replenishment message with structure including the mandatory / optional elements. If the validations fail, an error message / fault is returned to the invoking component.

4.  If Validation Type is specified and not 0, Balance Replenishment Proxy component invokes [performOrderFeasibility](../../fulfillment/Common Fulfillment/Order Validation.md#performorderfeasibility) operation of [Order Validation](../../fulfillment/Common Fulfillment/Order Validation.md) service to validate if the Plan Offering can be subscribed by the customer / Product(s).

5.  In case of successful validation of Balance Replenishment request, Order Validation component returns the response enriched with the retrieved information from [retrieveCustomerServiceConfiguration](../../inventory/Manage Customer Information/Customer Configuration Inquiry.md#retrievecustomerserviceconfiguration) operation of [Customer Configuration Inquiry](../../inventory/Manage Customer Information/Customer Configuration Inquiry.md) service. Balance Replenishment Proxy component enriches the retrieved information in the request message; otherwise an error message / fault is returned to the invoking component.

6.  If Channel Id not in [90: Paytm, ZO: PhonePe, 99: Online aggregator, 20: Self Care, 92: MyJio, 85: Jio Money, 95: IVR] (Configurable List) and EWallet Reservation Reference Id is not available

    1.  Balance Replenishment Proxy component invokes [updateEWalletBalance](../Manage EWallet/EWallet Management.md#updateewalletbalance) operation of [EWallet Management](../Manage EWallet/EWallet Management.md) service to reserve the amount in the merchant's wallet.  
        **Note:** specify characteristics[name = "SEND_NOTIFICATION"]/value = "N" to suppress the wallet deduction notification to dealers for balance replenishment.

    2.  If wallet reservation fails, an error message / fault is returned to the invoking component.

    3.  If the wallet reservation is successful, "EWallet Reservation Reference Id" is enriched in the request message.

7.  If Replenishment Order Validation is applicable and successful or not applicable,

    1.  Balance Replenishment Proxy component determines if message enrichment is required  
        **Note:** If following elements are not present, enrichment is required: Customer ID, Account, Service details

        1.  Balance Replenishment Proxy then puts the message to a messaging queue for processing by Balance Replenishment Enrichment component.

        2.  Balance Replenishment Enrichment component retrieves the message from the messaging queue.

        3.  If the request does not contains JMS Message Property FRC = F

            -   Balance Replenishment Enrichment component invokes [retrieveCustomerServiceConfiguration](../../inventory/Manage Customer Information/Customer Configuration Inquiry.md#retrievecustomerserviceconfiguration) operation of [Customer Configuration Inquiry](../../inventory/Manage Customer Information/Customer Configuration Inquiry.md) service to retrieve the subscriber details and enrich the message.  
                **Note:** retrieveCustomerServiceConfiguration is invoked for Service ID(s) as received from Service Consumer.  
                **Note:** specify /filterKey = CFS, includeAllServices = true to retrieve and enrich the data required – Customer ID, Account Information, Characteristic [Name = PRODUCT_STATE], Characteristic [Name = CUSTOMER_DETAILS] and CFS Details.  
                **Note:** Following are the characteristics enriched based on the information retrieved  
                name = PRODUCT_STATE  
                value = <Code from the lov: STATE\>  
                name = CUSTOMER_DETAILS  
                value = <Customer Name; Concat First Name, Middle Name, Last Name with a space\>  
                **Note:** Identify and use the Connectivity Account (Account Type = 1: CONNECTIVITY)  
                **Note:** Enrich the message with Service ID, Charging ID(s) and Service Type for all the services for the account for which request is submitted.  
                **Note:** Circle is an attribute of an Account.  
                **Note:** If jioroute is not received in the request, it is enriched along with the other values.

                ```title="Note:"
                Enrich Date of Birth, if required
                If /lineItems/commonDataList/characteristics/commonDataList[name = 'DOB']/value
				    does NOT exist and from retrieveCustomerServiceConfiguration; 
					/customer/dateOfBirth exist
                         
					specify /lineItems/commonDataList/characteristics/commonDataList[name = 'DOB']/value
                    = </customer/dateOfBirth>
                ```

        4.  Balance Replenishment Enrichment component persists the message to a messaging queue for further processing by Balance Replenishment component.

    2.  Balance Replenishment Proxy component then puts the message to a messaging queue for processing.

8.  An acknowledgment is then returned to the invoking component about successfully receiving the message.

9.  Balance Replenishment component retrieves the message from the messaging queue.

10.  Balance Replenishment component invokes handleNotification operation of Customer Notification service to submit a request for sending Transaction Acknowledgement notification to the customer.  
    **Note:** If characteristics [name = 'ADDL_NOTIFICATION_NUMBER']/value or characteristics [name = 'ADDL_NOTIFICATION_EMAIL']/value exists, notifications are also emitted to the specified destination Mobile Number and / or Email ID.

11.  Balance Replenishment component retrieves the transaction routing details (Endpoint URL) for the destination instance of Estel. Refer [Routing – Outbound – Service Component to route](../../appendices/Appendix G.md#routing-outbound-service-component-to-route) for details.

12.  Balance Replenishment component translates the request from CMM(Reliance SID) to the proprietary format of Estel and invokes topup operation of the Estel interface. Balance Replenishment receives the transaction confirmation after translating response message from proprietary format of Estel to the CMM(Reliance SID) format.  
    **Note:** Estel performs lookup based on the Merchant Code and enriches characteristics[name = MERCHANT_STATE] in the request message.  
    **Note:** Estel in-turn invokes [rechargeAccount](Refill Management.md#rechargeaccount) operation of [Refill Management](Refill Management.md) service

    ```title="Note:"
    If Plan Offering Type = 33: COMPOSITE VOUCHER,
        Estel retrieves Plan Offering details from local copy of Catalog and enriches in the response
		for following elements:
    
	    /return/amount = <Gross Amount of Plan Offering>
        /return/netAmount = <Net Amount of Plan Offering>
        /return/processingFee = <Processing Fees of Plan Offering>
        /return/taxesDeducted = <Total Tax deducted for Plan Offering>
        /return/taxInfo[]/taxName = <applicable Tax Name>
        /return/taxInfo[]/taxAmount = <applicable Tax Amount>
    ```

13.  If the Success response is received

    1.  If Plan Offering Details are not available

        1.  Balance Replenishment component retrieves the Plan Offering details from the Cache.

        2.  If Plan Offering is not identified, Balance Replenishment component invokes [queryPlanOffering](../../catalog/Manage Product Offering/Product Offering Inquiry.md#queryplanoffering) operation of [Product Offering Inquiry](../../catalog/Manage Product Offering/Product Offering Inquiry.md) service to retrieve the replenishment (top-up, recharge) offering details based on input unique ID.  
            **Note:** specify /planOffering/planID = <from request; /lineItems/commonDataList/productID\>, /filterCriterias[name = 'PLANSPECIFICATION']/value = <PLANSPECIFICATION\>

    2.  Balance Replenishment component invokes handleNotification operation of Customer Notification service to submit a request for sending Transaction Status notification to the customer.  
        **Note:** If characteristics [name = 'ADDL_NOTIFICATION_NUMBER']/value or characteristics [name = 'ADDL_NOTIFICATION_EMAIL']/value exists, notifications are also emitted to the specified destination Mobile Number and / or Email ID.  
        **Note:**
        if /planOffering/planType retrieved from cache for the Plan Offering = 33: COMPOSITE VOUCHER  
        specify /templateID = 165800,  
        /referenceID = <from request; /header/rechargeRefNo\>,  
        /payloadMessage/serviceId = <from response of retrieveCustomerServiceConfiguration; /customer/product[id = 'P10016']/identifier[category = '4' and subCategory = '2']/value\>,  
        /payloadMessage/attributes/commonDataList[name = 'TRANSACTION_ID']/value = <from request; /header/rechargeRefNo\>,  
        /payloadMessage/attributes/commonDataList[name = 'MSISDN']/value = <from response of retrieveCustomerServiceConfiguration; /customer/product[id = 'P10016']/identifier[category = '4' and subCategory = '2']/value\>,  
        /payloadMessage/attributes/commonDataList[name = 'GROSS_AMOUNT']/value = <from response of topup; /return/amount\>  
        /payloadMessage/attributes/commonDataList[name = 'QUANTITY']/value = <specify constant '1'\>  

    3.  If the Characteristic CALLBACK_REF_NOTIFY_URL is available in request (identified by /lineItems/commonDataList/characteristics/commonDataList[name = 'CALLBACK_REF_NOTIFY_URL']/value exists),

        1.  Balance Replenishment performs a lookup for the LOV MERCHANT_DESTINATION (data for which is cached on startup) based on the received merchantCode (identified from request message; /header/merchantCode).

        2.  If the destination messaging queue is retrieved for the merchantCode

            -   Retrieved destination messaging queue of notifyBalanceReplenishmentStatus is used to submit the request for sending notification to the service consumer.

        3.  Otherwise

            -   Default / existing messaging queue of notifyBalanceReplenishmentStatus is used to submit the request for sending notification to the service consumer.

        4.  Balance Replenishment component invokes [notifyBalanceReplenishmentStatus](#notifybalancereplenishmentstatus) operation of [Balance Replenishment](#balance-replenishment) service to submit a request for sending notification to the service consumer by persisting a message into the identified messaging queue.

    4.  If the (Merchant Contact Number is available in response) and (Channel != 99 : Online aggregator or 85 : Jio Money) [Configurable list of Channels], Balance Replenishment component invokes handleNotification operation of Customer Notification service to submit a request for sending notification to the retailer / merchant by persisting a message into a messaging queue.  
        **Note:** Merchant Contact Number indicates; initiated request is originated from RPOS.

14.  Otherwise (Failure response) [Refer Issue #6]

    1.  Balance Replenishment component determines the action to be performed for the specific error code.  
        **Note:** Error is identified by /statisticsInfo[PName='WALLET_BLOCK_STATUS']/PValue = RELEASED: Wallet is released  
        **Note:** Relay performed when /statisticsInfo[PName='WALLET_BLOCK_STATUS']/PValue = BLOCKED: Wallet is blocked  
        **Note:** Replay Policy for Balance Replenishment is: Subsequent to x automated retries after y duration, moves to Error (x = 3 & y = 2 minutes [Configurable])

    2.  Error:

        1.  Balance Replenishment component invokes handleNotification operation of Customer Notification service to submit a request for sending Transaction Status notification to the customer.  
            **Note:** Caters to the scenarios for failure in Voucher redemption, and exceeding the limit for balance replenishments as supported by OCS.  
            **Note:** If characteristics [name = 'ADDL_NOTIFICATION_NUMBER']/value or characteristics [name = 'ADDL_NOTIFICATION_EMAIL']/value exists, notifications are also emitted to the specified destination Mobile Number and / or Email ID.

        2.  If the Characteristic CALLBACK_REF_NOTIFY_URL is available in request,

            -   Balance Replenishment performs a lookup for the LOV MERCHANT_DESTINATION (data for which is cached on startup) based on the received merchantCode (identified from request message; /header/merchantCode).

            -   If the destination messaging queue is retrieved for the merchantCode

                -   Retrieved destination messaging queue of notifyBalanceReplenishmentStatus is used to submit the request for sending notification to the service consumer.

            -   Otherwise

                -   Default / existing messaging queue of notifyBalanceReplenishmentStatus is used to submit the request for sending notification to the service consumer.

            -   Balance Replenishment component invokes [notifyBalanceReplenishmentStatus](#notifybalancereplenishmentstatus) operation of [Balance Replenishment](#balance-replenishment) service to submit a request for sending notification to the service consumer by persisting a message into the identified messaging queue.

        3.  If the (Merchant Contact Number is available in response) and (Channel != 99: Online aggregator or 85: Jio Money) [Configurable list of Channels], Balance Replenishment component invokes handleNotification operation of Customer Notification service to submit a request for sending notification to the retailer / merchant by persisting a message into a messaging queue.  
            **Note:** Merchant Contact Number indicates; initiated request is originated from RPOS.

    3.  Replay – Automated: Balance Replenishment component persists the message to a messaging queue for message to be replayed.

    4.  Manual Intervention: Balance Replenishment component persists the message to a messaging queue for human intervention.  
        **Note:** A User Interface, developed using Business Events (Process Monitoring and Management) is used for browsing the queue, performing analysis and replay of messages.  
        **Note:** For Replay-Automated and Manual Intervention scenarios, Failed Message is enriched with the Transaction ID, received from the topup operation of Estel, before persisting in the messaging queue. The replayed Message specifies an ESTEL Transaction ID in the request. Message which is replayed, upon subsequent failure, is not updated with the new Transaction ID.  
        **Note:** Handling of Failure scenarios for the request is same; despite of the request origination channel.

Following is a textual walk-through of the Balance Replenishment – notifyBalanceReplenishmentStatus service.

1.  Balance Replenishment component consumes the message from the messaging queue.

2.  If Characteristics/Characteristic[Name = "CALLBACK_REF_NOTIFY_URL"]/Value begins with "jms"

    1.  Balance Replenishment component persists the message to the ReplyToQueue identified from the Value.

        **Note:** REST Gateway (SECO) invokes the API exposed by the third party aggregator based on the callbackReference/notifyURL specified in the request.
