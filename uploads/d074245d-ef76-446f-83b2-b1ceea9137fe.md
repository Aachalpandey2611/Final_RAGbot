Following is a textual walk-through of the Balance Recipient Maintenance:maintainRecipient operation.

1.  The consumer application (e.g. Self Care) invokes maintainRecipient operation of Balance Recipient Maintenance service to add or delete recipient.

2.  Balance Recipient Maintenance Proxy component performs the basic semantic validations such as compliance of recipient message with structure including the mandatory / optional elements.

3.  In case of successful validation of the message, Balance Recipient Maintenance Proxy component puts the message to a messaging queue for processing.

4.  An acknowledgment (including the Miscellaneous Requests Reference Number) is then returned to the invoking component about successfully receiving the recipient maintenance request (i.e. recipient registration or deregistration).

5.  Balance Recipient Maintenance component consumes the message from the messaging queue.

6.  Balance Recipient Maintenance component retrieves the transaction routing details (Endpoint URL) for the destination instance of SAP CRM. Refer [Routing – Outbound – Service Component to route](../../appendices/Appendix G.md#routing-outbound-service-component-to-route) for details.

7.  Balance Recipient Maintenance component translates the request from CMM (Reliance SID) to the proprietary message model of SAP and invokes the SAP CRM RFC ZCRM_BENEFICIARY_UPDATE to register or deregister the recipient.  
    **Note:** SAP CRM performs following validations prior to performing the registration of the recipients (considering in-progress registrations) and returns error when any of the validations fail.
    -  For a Donor, the max number of recipients is specified as Characteristic in request  
	   Name = MAX_RECIPIENT_CNT,  
	   Value = (from Plan Offering Characteristic)
    -  Recipient must not already be registered as a donor or recipient

    **Note:** Following are the error codes returned from SAP CRM
	
    -  697: Recipient &1 already registered/deregistered
    -  698: Recipient &1 already associated with other donor
    -  699: Recipient is already a donor
    -  700: Max limit of allowed recipient association reached
    -  701: Donor is already a recipient

8.  If an error is returned and error code in [697, 698, 699, 700, 701] and request NOT initiated from Channel IN [Configurable list of channel],
    1.  Balance Recipient Maintenance component invokes [handleNotification](../../common/Customer Communication/Customer Notification.md#handlenotification) operation of [Customer Notification](../../common/Customer Communication/Customer Notification.md) service to submit a request for sending notification to the Customer, by putting a message to the messaging queue.  
        **Note:** Configurable list of channel = [15: Campaign Fulfillment Platform]  
		**Note**: Notification Template Id = 205417  
        **Note:** Once the registration or deregistration of recipient is successful, SAP CRM invokes [notifyRecipient](Balance Recipient Publication.md#notifyRecipient) for processing the registeration or deregisteration of the recipient.

9.  If JMS Property JMSReplyTo exists,
    1.  Balance Recipient Maintenance component translates the message from the proprietary message model of SAP to CMM (Reliance SID) and persists the response message to the messaging queue based on the JMSReplyTo, to return the response to the originating component of the request.
