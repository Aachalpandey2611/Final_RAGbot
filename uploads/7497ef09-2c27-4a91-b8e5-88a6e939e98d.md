Following is a textual walkthrough of the Balance Replenishment Reversal Validation:performBalanceRecoveryFeasibility service.

1.  The consumer application invokes performBalanceRecoveryFeasibility operation of Balance Replenishment Reversal Validation service for feasibility to reverse the balance replenishment transaction.

2.  Balance Replenishment Reversal Validation component invokes [performReversalFeasibility](Integrated Recharge Reversal.md#performreversalfeasibility) operation of [Integrated Recharge Reversal](Integrated Recharge Reversal.md) service to validate the balance replenishment reversal.  
    **Note:** specify /customerId = <Customer Id\>

3.  If Balance Replenishment transaction cannot be reversed (identified by /resultStatus/status = 'FAILED')
    1.  return error, as received

4.  Balance Replenishment Reversal Validation component invokes [queryReplenishment](../Manage Invoice/Bill Inquiry.md#queryreplenishment) operation of [Bill Inquiry](../Manage Invoice/Bill Inquiry.md) service to retrieve the status of selected balance replenishment transaction.  
    **Note:** specify /customerId = <Customer Id\>, /replenishmentOrderRefNo = <selected transaction for reversal; /replenishmentOrderRefNo/replenishmentOrderRefNo\>  
    **Note:** Plan Offering(s) from queryReplenishment are identified as /replenishmentOrder/replenishmentOrderLineItem[]/planOfferingId

5.  Balance Replenishment Reversal Validation component performs following validations:
    1.  All Status of the transaction is in configurable list [0: ACTIVE, 1: QUEUED]
        -  If Status of transaction(s) (identified by /replenishmentOrder/replenishmentOrderLineItem[]/status) is NOT in configurable list [0, 1], return error 2021: Plan Offering not active or in queue is not eligible for reversal.

6.  Balance Replenishment Reversal Validation component retrieves Plan Offering details from EDIF for the retrieved list of Plan Offering(s) for the balance replenishment transaction.  
    **Note:** specify /planOffering[]/planID = <from response of queryReplenishment; /replenishmentOrder/replenishmentOrderLineItem[]/planOfferingId\>  
    **Note:** Data elements are retrieved using native connection to EDIF (JDBC)  
    **Note:** Retrieve following details of the Plan Offering(s)
    -  PlanOffering.type
    -  PlanOffering.subType
    -  Any PlanSpecification.serviceSpecificationId = F30003: International Roaming (via. PlanOfferingToPlanSpecMapping) for the Plan Offering
    -  Any PlanSpecification.serviceSpecificationId = F30002: ISD (via. PlanOfferingToPlanSpecMapping) for the Plan Offering
    -  Any PlanSpecification.serviceType = Z0072: Netflix exists (via. PlanOfferingToPlanSpecMapping) for the Plan Offering (Optional)  
       **Note:** returned for Plan Offering, only if Netflix is bundled with the Plan Offering

7.  Balance Replenishment Reversal Validation component performs following validations:
    1.  All Plan Offering Type must be one of the following (configurable list):
        -  1: CONNECTIVITY RECHARGE
        -  7: CONNECTIVITY + CONTENT/OTT RECHARGE
        -  13: TOPUP
        -  25: NONMONETARY TOPUP
        -  26: NONMONETARY TOPUP-INHERITING VALIDITY ONLY
        -  If any Plan Offering Type is NOT in configurable list [1, 7, 13, 25, 26], return error 2023: Plan Offering Type is not eligible for reversal.
    2.  Balance Replenishment transactions are NOT for Plan Offerings of Speed Booster or Validity Booster
        -  Any Plan Offering Sub type must NOT be one of the following (configurable list):
           -  5: BW boost
           -  12: Validity Boost
        -  If any Plan Offering subType is in configurable list [5, 12], return error 2024: Speed Booster or Validity Booster Transaction is not eligible for reversal.
    3.  Selected Balance Replenishment is in Stack and has bundled Netflix Plan which is NOT upgraded using Netflix Flexi Top-Up
        -  For any PlanSpecification of Plan Offering(s), serviceType = Z0072: Netflix exists
        -  If Bundled NetfixOfferId exists (from cache; serviceType = Z0072: Netflix exists) and Plan Offering is in Stack (identified by /replenishmentOrder/replenishmentOrderLineItem/status = 0: ACTIVE)
            *  Invoke retrieveDigitalServiceByPlanOffering operation of Digital Service Configuration Inquiry service to retrieve the subscribed digital services by Plan Offering.  
              **Note:** specify /orderReferenceNumber = <reversal Order Reference Number\>, /customer/customerId = <from request message; /customerId\>
            *  If Netflix is upgraded using Netflix Flexi Topup (identified from response of retrieveDigitalServiceByPlanOffering; /customers/planOfferings[stackedQueued = '0']/digitalServices[serviceType = 'Z0072']/characteristics[name = 'LAST_TRANS_REF_NO'] exists)
                -  return error 2025: Netflix subscription is already upgraded; Transaction is not eligible for reversal

8.  If bUPCGENDTTM_EDIF = Y  
    **Note:** bUPCGENDTTM_EDIF is a local configuration to determine "Last UPC Generation DateTime" from EDIF
    1.  Balance Replenishment Reversal Validation component invokes [queryCustomer](../../inventory/Manage Customer Information/Customer Inquiry.md#querycustomer) operation of [Customer Inquiry](../../inventory/Manage Customer Information/Customer Inquiry.md) service to retrieve the Customer details.  
        **Note:** specify /customerId = </customerId\>, /filterKeyList/filterKey = <PRODUCT\>
    2.  Balance Replenishment Reversal Validation component determines "Last UPC Generation DateTime" (upcGenDTTM) from response of queryCustomer; /product[productCode = 'P10016']/mnpDetails/upcGeneratedDate
    3.  If upcGenDTTM exist and upcGenDTTM \>= blockTimeStart  
        **Note:** If UPC Code is generated after or upto 7 days before Balance Replenishment transaction, reversal is NOT allowed  
        **Note:** blockTimeStart = <from response of queryReplenishment; /replenishmentOrder/transDateTime – REVERSAL_BLOCK_PERIOD\> where REVERSAL_BLOCK_PERIOD = 10080 mins (configurable)
        -  return error 2028: Recharge reversal is not allowed for in-progress port out request

9.  Otherwise
    1.  Balance Replenishment Reversal Validation component invokes [findCustomerProblem](../../assurance/Manage Customer Problem Information/Customer Problem Inquiry.md#findcustomerproblem) operation of [Customer Problem Inquiry](../../assurance/Manage Customer Problem Information/Customer Problem Inquiry.md) service to retrieve any mute Customer Problem created after or upto 7 days before Balance Replenishment transaction selected for reversal.  
        **Note:** blockTimeStart = <from response of queryReplenishment; /replenishmentOrder/transDateTime – REVERSAL_BLOCK_PERIOD\> where REVERSAL_BLOCK_PERIOD = 10080 mins (configurable)  
        **Note:** specify /customerId = </customerId\>, /problemType = <ZNPS\>, /dateRange/dateFrom = <blockTimeStart\>, /dateRange/dateTo = <System DateTime\>, /paging/pagesize = <20\>, /paging/offset = <0\>
    2.  If Customer Problem is NOT retrieved (identified by /resultStatus/status = "FAILED" and /resultStatus/errorCode = "017")
        -  continueOn i.e. Error to be suppressed, no need to log and continue processing.
    3.  Else If Customer Problem(s) is retrieved (identified by /resultStatus/status = "SUCCESS")
        -  Balance Replenishment Reversal Validation component iterates through the list of retrieved customerProblem  
           **Note:** If UPC Code is generated after or upto 7 days before Balance Replenishment transaction, reversal is NOT allowed
           -  If /subSubCategory = UPC_SUBSUBCATG (configurable; C706) and /raisedDate \>= blockTimeStart
              -  return error 2028: Recharge reversal is not allowed for in-progress port out request

10.  Balance Replenishment Reversal Validation component returns the response to the invoking component.
