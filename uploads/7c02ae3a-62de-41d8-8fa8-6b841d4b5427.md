Following is a textual walkthrough of the Balance Replenishment Reversal:recoverBalance service.

**Note:** Only one Order Line Item is supported.

1.  The consumer application invokes recoverBalance operation of Balance Replenishment Reversal service for reversal of balance replenishment transaction of pre-paid balances for an account.

2.  Balance Replenishment Reversal Proxy component performs the basic semantic validations such as compliance of Balance Replenishment Reversal message with structure including the mandatory / optional elements. If the validations fail, an error message / fault is returned to the invoking component.

3.  Balance Replenishment Reversal Proxy component invokes [maintainReversalStatus](Integrated Recharge Reversal.md#maintainreversalstatus) operation of [Integrated Recharge Reversal](Integrated Recharge Reversal.md) service to validate and initiate the balance replenishment reversal.  
    **Note:** specify /status = 1: Reversal Initiated

4.  If request for balance recovery is initiated successfully (/resultStatus/status = 'SUCCESS')
    1.  Balance Replenishment Reversal Proxy component persists the message into the corresponding messaging queue to be processed by Balance Replenishment Reversal component.
    2.  The acknowledgement message including the Balance Recovery Reference Number (as received from Service Consumer) is then returned to the invoking component.

5.  Otherwise
    1.  If /resultStatus/errorCode NOT IN configurable list [2011, 2012, 2014, 2015, 2016]
        1.  Balance Replenishment Reversal Proxy component invokes [maintainReversalStatus](Integrated Recharge Reversal.md#maintainreversalstatus) operation of [Integrated Recharge Reversal](Integrated Recharge Reversal.md) service to cancel the initiated balance replenishment reversal.  
           **Note:** specify /status = 3: Reversal Cancelled / Failed
    2.  Balance Replenishment Reversal Proxy component returns the error message / fault to the invoking component.

6.  Balance Replenishment Reversal component then consumes the message from the messaging queue.

7.  Balance Replenishment Reversal component invokes [handleNotification](../../common/Customer Communication/Customer Notification.md#handlenotification) operation of [Customer Notification](../../common/Customer Communication/Customer Notification.md) service to submit a request for sending transaction acknowledgement notification to the customer.  
    **Note:** Specify /notification/templateID = <410717\>,  
    /notification/referenceID = <from request message; /orderRefNo\>  
    Specify Service Id and Circle to submit the notification in English and Vernacular language for the Circle  
    /notification/payloadMessage/serviceID= <from request message; /lineItem/planOfferings/accounts/products[services/serviceType = 'Z0003']/serviceId\>  
    /notification/payloadMessage/attributes/attribute[name='Pref_CIRCLE_ID']/value = <from request message; /lineItem/planOfferings/accounts/circleId\>  
    /notification/payloadMessage/attributes/attribute[name = 'REFUND_AMOUNT']/value = <from request message; /paymentDetails/totalAmountPayable\>  
    /notification/payloadMessage/attributes/attribute[name = 'MSISDN']/value = <from request message; /lineItem/planOfferings/accounts/products[services/serviceType = 'Z0003']/serviceId\>  
    /notification/payloadMessage/attributes/attribute[name = 'TXN_ID']/value = <from request message; /lineItem/adjustmentDetail/rechargeRefNo\>

8.  Balance Replenishment Reversal component invokes [getTransactionRefNumber](../../common/Manage Identifiers/Identifier Management.md#gettransactionrefnumber) operation of [Identifier Management](../../common/Manage Identifiers/Identifier Management.md) service to generate and fetch a Transaction Reference Number.  
    **Note:** specify /transactionType = "PP" to generate a transaction reference number

9.  Balance Replenishment Reversal component invokes [rechargeAccount](Refill Management.md#rechargeaccount) operation of [Refill Management](Refill Management.md) service to process the reversal transaction in corresponding OCS.

10.  If the Success response is received
     1.  If Payment Bifurcation received in request (identified by /paymentDetails/paymentBifurcation exists)
         1.  If Plan Offering type = 13: TOPUP (identified by /lineItem/planOfferings/planOfferingType = '13')
             -  If Gross Amount for refund (from response of rechargeAccount; /lineItem/grossAmount) = Payment Amount (/paymentDetails/totalAmountPayable)  
               **Note:** Indicates complete amount is recovered
                 *  refundPaymentBifurcation = /paymentDetails/paymentBifurcation  
                  **Note:** Refund is provided to each Payment Mode as received in Payment Bifurcation.
             -  Otherwise
                 *  Determines following Payment Amount from request (if not received, consider zero)
                     -  initialize forfeitThresholdAmount = 0
                     -  Iterate through each /paymentDetails/paymentBifurcation[]
                         -  if /paymentDetails/paymentBifurcation[i]/modeOfPayment IN [LST_PAYMENT_MODE_FORFEIT_THREHOLD]  
						     where LST_PAYMENT_MODE_FORFEIT_THREHOLD = [56, 57, 58] (configurable list)
                             -  forfeitThresholdAmount = forfeitThresholdAmount + /paymentDetails/paymentBifurcation[i]/amount  
                           **Note:** forfeitThresholdAmount = Cashback Amount + eCoupon Amount + Instant Discount Amount
                         -  If /paymentDetails/paymentBifurcation[isPrimary = '1']
                             -  Payment Amount (Primary) = /paymentDetails/paymentBifurcation[i]/amount
                *  If Gross Amount \>= forfeitThresholdAmount  
                  **Note:** To identify the scenario for forfeit of the Voucher Amount (Cashback or eCoupon) when Gross Amount is less than the Voucher + Instant Discount  
                  **Note:** Any Deduction is first made from MyJio Store Credit (Secondary), if available and subsequently from Payment Mode (Primary).
                    -  If MyJio Store Credit (Secondary) is used (identified by /paymentDetails/paymentBifurcation[not(isPrimary = '1')]/modeOfPayment = '42')
                        -  If Gross Amount \> Payment Amount (Primary) + forfeitThresholdAmount  
                           **Note:** Deduction made from MyJio Store Credit (Secondary)
                             -  Refund Amount for MyJio Store Credit (Secondary) = Gross Amount - (Payment Amount (Primary) + forfeitThresholdAmount)
                             -  refundPaymentBifurcation = update Refund Amount for MyJio Store Credit (Secondary); /paymentDetails/paymentBifurcation[modeOfPayment = '42' and not(isPrimary = '1')]/amount = <Refund Amount for MyJio Store Credit (Secondary)\>
                        -  Otherwise  
                            **Note:** Deduction made from MyJio Store Credit (Secondary) and then Payment Mode (Primary)
                            -  Refund Amount for Payment Mode (Primary) = Gross Amount - forfeitThresholdAmount
                            -  refundPaymentBifurcation = update the Refund Amount for Payment Mode (Primary); /paymentDetails/paymentBifurcation[isPrimary = '1']/amount = <Refund Amount for Payment Mode (Primary)\>
                            -  refundPaymentBifurcation = remove paymentBifurcation for MyJio Store Credit (Secondary); /paymentDetails/paymentBifurcation[modeOfPayment = '42' and not(isPrimary = '1')]
                    -  Otherwise
                        -  Refund Amount for Payment Mode (Primary) = Gross Amount - forfeitThresholdAmount
                        -  refundPaymentBifurcation = update the Refund Amount for Payment Mode (Primary); /paymentDetails/paymentBifurcation[isPrimary = '1']/amount = <Refund Amount for Payment Mode (Primary)\>
                *  Otherwise  
                    **Note:** Forfeit the complete amount and do not refund anything.  
                    **Note:** Scenario for forfeit of the Voucher Amount (Cashback or eCoupon or JioMart) when Gross Amount is less than the Voucher + Instant Discount
                    -  refundPaymentBifurcation = update the Refund Amount for Payment Mode (Primary) = 0; /paymentDetails/paymentBifurcation[isPrimary = '1']/amount = <0\>
                    -  refundPaymentBifurcation = remove all paymentBifurcation other than primary; /paymentDetails/paymentBifurcation[not(isPrimary = '1')]
                    -  Override Gross Amount = 0; /lineItem/grossAmount = 0
         2.  Otherwise
            -  refundPaymentBifurcation = /paymentDetails/paymentBifurcation  
               **Note:** Refund is provided to each Payment Mode as received in Payment Bifurcation.
         3.  If received Payment Mode = 42: MyJio Store Credit (identified by /paymentDetails/modeOfPayment = '42')  
            **Note:** Customer has chosen to perform refund to MyJio Store Credit
            -  Update Payment Mode (Primary) to MyJio Store Credit; /paymentDetails/paymentBifurcation[isPrimary = '1']/modeOfPayment = '42'
     
	 2.  Balance Replenishment Reversal component translates the request from CMM (Reliance SID) to the proprietary message model of Estel.  
         
		```title="Note:"
		If Plan Offering type = 13: TOPUP, 
		     specify the gross refund amount received from OCS / adjusted amount  
             if /lineItem/planOfferings/planOfferingType = '13',  
                 specify /topupInfoRequest/statisticsInfo[pName = 'RECOVER_BALANCE']/pValue = 
				 <Gross refund amount received in response of rechargeAccount; /lineItem/grossAmount>  
		```
         
		 **Note:** specify refundPaymentBifurcation as paymentBifurcation in the request to Estel, if determined
     
	 3.  Balance Replenishment Reversal component persists the message to a messaging queue to be consumed by Zonal Instance of Estel – Retail.  
         **Note:** Messaging Queue for the Zonal Instance of Estel – Retail is identified based on jiodealerroute.  
         **Note:** Estel uses Recharge Order Reference Number (of the Transaction for which recovery transaction is submitted) and Customer Id for retrieving the original transaction and the determine the required monetary values for reversal. The reversal amount is same as the amount from the original transaction.  
         ```title="Note:"
		 If recovery amount is received; applicable for TOPUP reversal transactions 
		 (identified by /topupInfoRequest/statisticsInfo[pName = 'RECOVER_BALANCE']/pValue exists), 
		     Estel uses the received Gross Amount, calculates tax and net amount.  
         Otherwise 
		     Estel uses Recharge Order Reference Number (of the Transaction for which recovery transaction 
			 is submitted) and Customer Id for retrieving the original transaction and the determine 
			 the required monetary values for reversal. 
			 The reversal amount is same as the amount from the original transaction  
         ```
		 ```title="Note:"
		 If Payment Bifurcation is received (identified by /topupInfoRequest/paymentBifurcation exists), 
		     Estel uses Payment Mode(s) from Payment Bifurcation 
			 (/topupInfoRequest/paymentBifurcation[]/modeOfPayment) 
			 received in request while creating the Credit Note  
         Otherwise 
		     Estel uses the Payment Mode (identified by /topupInfoRequest/paymentMode) 
			 received in request while creating the Credit Note.  
         ```
		 **Note:** Estel persists the determined refund amount details such as Gross Amount, NetAmount, List of Tax and Processing Fees in the local data store.
     4.  Balance Replenishment Reversal component invokes [maintainReversalStatus](Integrated Recharge Reversal.md#maintainreversalstatus) operation of [Integrated Recharge Reversal](Integrated Recharge Reversal.md) service to complete the balance replenishment reversal.  
         **Note:** specify /status = 2: Reversal Completed
     5.  If the Characteristic CALLBACK_REF_NOTIFY_URL is available in request (identified by /lineItem/characteristics[name = 'CALLBACK_REF_NOTIFY_URL']/value exists),
         -  Balance Replenishment Reversal component invokes [notifyBalanceRecoveryStatus](#notifybalancerecoverystatus) operation of [Balance Replenishment Reversal](#) service to submit a request for sending notification to the service consumer by persisting a message into the messaging queue.  
            **Note:** specify refundPaymentBifurcation as paymentBifurcation in the request, if determined

11.  Otherwise (Failure response)
     1.  Balance Replenishment Reversal component determines the action to be performed for the specific error code – classify as Business Exception or System Exception.
     2.  If Business Exception
         1.  Balance Replenishment Reversal component invokes [maintainReversalStatus](Integrated Recharge Reversal.md#maintainreversalstatus) operation of [Integrated Recharge Reversal](Integrated Recharge Reversal.md) service to cancel the balance replenishment reversal.  
             **Note:** specify /status = 3: Reversal Cancelled / Failed
         2.  If the Characteristic CALLBACK_REF_NOTIFY_URL is available in request (identified by /lineItem/characteristics[name = 'CALLBACK_REF_NOTIFY_URL']/value exists),
            -  Balance Replenishment Reversal component invokes [notifyBalanceRecoveryStatus](#notifybalancerecoverystatus) operation of [Balance Replenishment Reversal](#) service to submit a request for sending notification to the service consumer by persisting a message into the messaging queue.
     3.  Otherwise (System Exception)
         1.  Balance Replenishment Reversal component persists the message to a messaging queue for message to be replayed.  
             **Note:** Replay Policy for Balance Replenishment Reversal is: subsequent to x automated retries after y duration, moves to manual intervention (x = 3 & y = 5 minutes [Configurable]).  
             **Note:** A User Interface, developed using Business Events (Process Monitoring and Management) is used for browsing the queue, performing analysis and replay of messages.
